# План интеграции LLM в рой (для реализации другой сессией)

> Инфраструктурная часть (Docker/Ollama на Windows-ноде winpc) уже поднята и
> проверена руками (SSH) отдельной сессией 2026-07-23. Этот документ описывает,
> что нужно сделать **в коде sa-home-bot**, чтобы LLM стала обычным умением
> роя. Бот (`telegram-bot`, `bot/handlers/*`) сознательно НЕ трогался —
> по договорённости с пользователем это делает другая сессия/агент. Здесь —
> только служба `llm` и её подключение к ноде.

## 0. Ключевое архитектурное ограничение (прочитать первым)

**Никакого прямого HTTP/сетевого обращения между машинами.** Пользователь
явно ограничил дизайн: вызов LLM должен идти строго по существующему
протоколу роя (`PROTOCOL.md`), тем же путём, что и любое другое умение —
`bot → своя нода → (proto, TCP+token) → нода winpc → служба llm → Ollama`.
Никаких новых HTTP-эндпоинтов, наружу смотрящих портов и т.п. — это
дублировало бы то, что протокол роя уже даёт («спроси любого»,
ARCHITECTURE §11 п.2).

Единственное HTTP-обращение во всей цепочке — **локальное**: служба `llm`
дёргает Ollama по `http://127.0.0.1:11434` **на той же машине**, где сама
служба работает (winpc). Это не сетевой вызов в смысле роя, а обычный
localhost loopback, как `apps` дёргает `systemctl` локально.

Порт 11434 Ollama **сознательно не пробрасывается** ни в LAN, ни в tailnet —
firewall/portproxy правила заводились и тут же откачены. Наружу эта машина
ничего не слушает, кроме уже существующего proto-порта ноды (`8710`/`8711`
из её `[node]`/`[monitor]` секций).

## 1. Что уже готово на стороне Windows-ноды (winpc, 100.78.225.83)

- Docker Desktop **не используется** (машина — Windows 10 Enterprise LTSC
  2021, build 19044, без Microsoft Store; GUI-инсталлятор неудобно ставить
  через SSH-автоматизацию). Вместо него — **Docker Engine внутри
  WSL2-дистрибутива**, поднятого чистым `wsl --import`.
- Дистрибутив **`Docker`** (имя в `wsl -l -v`), Ubuntu 24.04 (base, минимальный
  `ubuntu-base` тарбол Canonical), **весь целиком лежит на `D:\WSL\Docker`**
  (vhdx-диск), НЕ на `C:` (там было всего 10GB свободных — по требованию
  пользователя все данные держим на D).
  - Внутри дистрибутива включён systemd (`/etc/wsl.conf`: `[boot]
    systemd=true`) — потребовалось поставить современный WSL-пакет
    отдельно (см. ниже), у inbox-`wsl.exe` этой LTSC-машины systemd не
    поддерживался.
  - Docker Engine (28.x/29.x, официальный репозиторий `download.docker.com`)
    + `nvidia-container-toolkit`, `docker.service` включён и активен через
    systemd.
  - GPU-проброс в контейнеры подтверждён: `docker run --gpus=all ...
    nvidia-smi` внутри `Docker`-дистрибутива видит RTX 3060 Ti напрямую —
    отдельный NVIDIA-драйвер внутри WSL не нужен, драйвер Windows (610.74)
    сам экспортирует GPU в WSL2.
- **Модель современного WSL** (`microsoft/WSL` релиз 2.7.10, MSI-инсталлятор
  с GitHub, Store не нужен) поставлена отдельно поверх inbox-версии — это
  дало systemd-поддержку и рабочий `wsl --version/--status`. После установки
  этого пакета в системе появляется новая служба **`WSLService`**, старая
  **`LxssManager`** становится неактуальна (Stopped) — это ожидаемо, не баг.
- Контейнер **`ollama`** (`ollama/ollama:latest`, движок 0.32.1):
  ```
  docker run -d --gpus=all -v /mnt/d/OllamaData:/root/.ollama \
    -p 11434:11434 --restart unless-stopped --name ollama ollama/ollama
  ```
  - Данные модели — на `D:\OllamaData` (внутри WSL смонтирован как
    `/mnt/d/OllamaData`), а не в volume внутри vhdx — так они видны и
    управляемы напрямую с Windows-стороны при необходимости.
  - `--restart unless-stopped` — контейнер поднимется сам, как только
    поднимется Docker-демон внутри дистрибутива (см. открытый вопрос
    в §4 про автостарт самого WSL-дистрибутива при загрузке Windows).
- Модель **`qwen2.5:7b`** (4.7GB, Q4-квантование по умолчанию из Ollama
  library) уже стянута. Подтверждён **полный GPU-оффлоад: 29/29 слоёв**
  (лог `load_tensors: offloaded 29/29 layers to GPU`), используется ~4.5GB
  VRAM из 8GB (буфер модели 4168MB + KV-cache 224MB + compute-буферы
  ~154MB) — есть запас под контекст побольше или вторую мелкую модель
  параллельно, если понадобится.
  - `OLLAMA_KEEP_ALIVE` по умолчанию `5m` — модель выгружается из VRAM
    после 5 минут простоя, повторная загрузка в тесте заняла ~600ms
    (`load_duration=595166778` ns) — не мгновенно, но и не критично; при
    желании поднять/убрать через переменную окружения контейнера.
  - Контекст по умолчанию — `default_num_ctx=4096` (авто-подбор от VRAM,
    `OLLAMA_CONTEXT_LENGTH` не задавался).
- Доступность API проверена **только с самой машины**:
  `http://127.0.0.1:11434/api/version` и `/api/generate` отвечают корректно
  и с Windows-хоста (через встроенный loopback-форвардинг WSL2), и изнутри
  WSL. Снаружи (LAN/tailnet) — недоступно, и это осознанный выбор (см. §0).
- Ручной консольный доступ для человека (не для роя, просто пощупать
  модель руками) настроен отдельно, через SSH на саму машину:
  ```bash
  ssh -t User@192.168.0.105 "wsl -d Docker -u root -- docker exec -it ollama ollama run qwen2.5:7b"
  ssh -t User@192.168.0.105 "wsl -d Docker -u root -- /root/.local/bin/oterm"   # TUI, oterm 0.20.0
  ```
  (`-t` обязателен — интерактивный терминал через несколько уровней
  ssh→wsl→docker exec). Логин на машине — `User` (не `sevboa`!), ключ и
  алиасы `winpc`/`desktop-ie953ua` уже есть в `~/.ssh/config` на alfred.
  На момент написания короткая LAN-запись `192.168.0.105` использовалась
  как обходной путь, пока Tailscale на машине не поднимался (см. §5) —
  для роя это не нужно, только для ручного дебага человеком.

## 2. Что нужно сделать в коде sa-home-bot

Служба `llm` — по образцу `apps`/`torrents` (см. `src/sa_home_bot/apps/`).
Точки, которые реально нужно тронуть (проверено чтением текущего кода):

### 2.1 Новый пакет `src/sa_home_bot/llm/`

- `service.py` — `LlmService`, реализует протокол `ServiceHandler`
  (см. `apps/service.py` как образец): `describe()`, `get_state()`,
  `run_command(action, args)`.
  - `describe()` → одно действие `ask`:
    ```json
    {"id": "ask", "title": "Спросить нейросеть", "params": [
      {"name": "prompt", "type": "string", "required": true, "title": "Вопрос"}
    ]}
    ```
    Возможно, второе read-only действие `list_models` (без параметров) —
    отдать список доступных моделей (`GET /api/tags`), на будущее если моделей
    станет несколько.
  - `run_command("ask", {"prompt": ...})` → HTTP POST на
    `http://127.0.0.1:11434/api/generate` (или `/api/chat`, если нужна
    история диалога — на первой итерации `/api/generate` со `stream: false`
    достаточно), таймаут заметно больше, чем у типичных «быстрых» действий
    ноды (см. открытый вопрос в §3 про таймауты клиента протокола).
    Возвращает `{"response": str, "model": str, "duration_ms": int}`.
  - HTTP-клиент — простой `urllib`/`httpx`/`aiohttp` (смотри, что уже есть
    в зависимостях пакета — вероятно, стоит переиспользовать то, что уже
    тянет `pyproject.toml`, а не добавлять новую тяжёлую зависимость).
  - Ошибки: сеть/таймаут/Ollama вернул не-200 → `ProtoError(ERR_INTERNAL, …)`
    (по аналогии с `apps/service.py::_run_systemctl`).
  - **Прогрев WSL на старте.** Служба `llm` на этой ноде физически работает
    как нативный Windows-процесс (весь sa-home-bot ставится на Windows
    через pipx под LocalSystem — см. `deploy/config-windows.example.toml`),
    а НЕ внутри WSL. Значит запуск самого sa-home-bot НЕ поднимает
    WSL-дистрибутив `Docker` автоматически — WSL стартует только когда
    что-то реально вызывает `wsl.exe` для этого дистрибутива. Если Windows
    перезагрузили, а nobody не зашёл и не подёргал `wsl`, то `Docker`
    (и, соответственно, `dockerd`/`ollama`) не поднимется сам, и первый
    `ask` упадёт с connection refused.
    **Нужно на старте `LlmService` (или лениво на первый вызов)**
    выполнить `subprocess` вызов `wsl -d Docker -u root -- true` (можно
    через `asyncio.create_subprocess_exec`, как это уже делает `apps`),
    дождаться его завершения — это разбудит WSL-VM, а
    `--restart unless-stopped` у контейнера ollama сам поднимет сервис
    следом, как только `dockerd` внутри стартанёт через systemd.
    Альтернатива/дополнение — Windows Scheduled Task «At startup» вне
    Python-кода, которая делает то же самое независимо от бота; решить
    на месте, что удобнее поддерживать.
- `app.py` — сборка: `LlmService` + `ProtoServer` на своём сокете
  (по аналогии с `apps/app.py`, если он есть, либо с тем, как собирается
  `monitor`/`torrents`).

### 2.2 `config.py`

Добавить `LlmConfig` (по образцу `AppConfig`/секций `monitor`/`apps`):

```toml
socket = "tcp://127.0.0.1:8712"   # свой proto-порт, следующий свободный после apps/torrents
```

Добавить обработку `--service llm` (по аналогии с существующими

### 2.6 Конфиги нод

- `config.example.toml` (Linux, справочный) — добавить пример секции
  `[llm]` для полноты (даже если реальная LLM стоит только на winpc —
  секция должна существовать в схеме `Settings`, иначе `--check-config`

- `deploy/config-windows.example.toml` и **реальный
  `C:\ProgramData\sa-home-bot\config.toml` на winpc** — добавить:
  ```toml

  [llm]
  ```

## 3. Открытые вопросы / на что обратить внимание при реализации

1. **Таймаут ProtoClient/ProtoServer.** Существующие действия
   (`scan_now`, `start/stop/restart` юнитов) — секунды. Генерация LLM на
   этой видеокарте для короткого ответа — доли секунды - пара секунд
   (после прогрева), но при длинном промпте/большом контексте может уйти
   заметно дальше. Проверить, есть ли в `proto/client.py`/`proto/server.py`
   захардкоженный таймаут на запрос-ответ, и если да — параметризовать
   его для конкретно этого действия, а не завышать глобально.
2. **Холодный старт после ребута Windows** (см. §2.1 про прогрев WSL) —
   решить, встроенный ли это в `LlmService`, или отдельная Windows-задача
   планировщика. Автовход (autologin) в Windows пользователь рассматривает
   отдельно (для Tailscale) — если сделает, заодно снимет и часть этой
   проблемы, но полагаться на это не стоит, т.к. это его отдельное решение
   вне зависимости от данного плана.
3. **Одна модель — один процесс.** Пока только `qwen2.5:7b`. Если в
   будущем появится вторая (в т.ч. на другой ноде) — `describe.actions`
   должен явно указывать, какая нода/модель отвечает (правило §11 п.5
   ARCHITECTURE — «одно умение, много исполнителей», роутинг по умолчанию
   на «свою» ноду или первую живую).
4. **Права и подписки.** Если/когда бот (другая сессия) заведёт команду
   `/ask`, право на неё по аналогии с `apps` должно быть вида `ask@llm` в
   `allowed_commands` подписки — сама служба `llm` авторизацией не
   занимается, это забота бота (см. AUTHORIZATION.md §3.3).

## 4. Как проверить, что всё работает (после реализации кода)

С любой ноды роя (например, alfred) через `nodectl` или напрямую через
`ProtoClient` с адресом `dst={"node": "winpc", "service": "llm"}`:
```
command: ask, args: {"prompt": "Привет, как дела?"}
```
должно вернуть осмысленный ответ за разумное время. Если winpc недоступна
(спит/офлайн) — по правилам роя это `unavailable`, честный быстрый отказ,
а не зависание (ARCHITECTURE §11 п.4).

## 5. Побочные заметки, не по теме кода, но важные для контекста

- SSH на winpc: логин **`User`**, не `sevboa`. Конфиг `~/.ssh/config` на
  alfred уже содержит алиасы `desktop-ie953ua`/`winpc` → `100.78.225.83`
  (через tailscale) с этим логином.
- Обнаружено: **Tailscale на этой Windows-машине поднимается только после
  интерактивного логина пользователя** (в отличие от SSH-сервера, который
  стартует как служба независимо от сессии) — после `shutdown /r` машина
  какое-то время недоступна по tailscale, хотя уже загрузилась и отвечает
  по SSH через обычный LAN IP. Пользователь рассматривает автовход
  (autologin) в Windows как решение — на момент написания плана ещё не
  включён. Если для роя (presence/heartbeat, ARCHITECTURE §11 п.4) важно,
  чтобы нода была видна сразу после перезагрузки хоста, эта деталь
  релев. Пароль на машине стоит в первую очередь ради RDP.
- В процессе установки несколько раз ловили зависание `wsl.exe`/
  `LxssManager` (сервис зависал в `StopPending`, новые вызовы `wsl` висли
  бесконечно) — лечилось только полной перезагрузкой Windows. Если при
  дальнейшей работе с этой нодой `wsl`-команды снова начнут зависать —
  это известный класс проблем этой конкретной машины (старая инсталляция
  WSL на LTSC), не повод подозревать код sa-home-bot.
- Машина периодически уходила в Windows Sleep, что рвало SSH/Tailscale и
  прерывало долгую операцию (докачку модели) без возможности резюме —
  Ollama просто отбрасывал недокачанный blob. **Сон на AC отключён**
  (`powercfg /change standby-timeout-ac 0`) — это было сделано в ходе
  данной сессии и относится к машине в целом, не только к LLM: без этого
  нода `winpc` не отвечает по протоколу роя, пока не проснётся, что
  ломает предположение ARCHITECTURE §11 п.4 «спящая не-24/7 нода — норма»
  (там про честную деградацию доступности умения, а не про то, что нода
  ГПУ-сервера должна засыпать сама по таймеру простоя).
- В библиотеке Ollama на 2026-07 доступен **Qwen3** (новее Qwen2.5,
  включая гибридные thinking/non-thinking режимы и MoE-веса `30b-a3b`).
  Пользователь пробовал `qwen3:14b` (9.3GB, Q4) — Ollama сам разбивает
  слои между GPU/CPU при нехватке VRAM: на этой видеокарте (8GB) вошло
  26 из 41 слоя, ~9.3 ток/с. Thinking-режим (включён по умолчанию) сильно
  раздувает время ответа скрытыми рассуждениями (поле `thinking` в
  ответе `/api/generate`) — с `"think": false` в запросе тот же ответ
  получается за ~10с вместо ~30с при той же скорости ток/с. Если для
  какого-то умения (в т.ч. будущего `/ai`, см. §6) важна отзывчивость —
  стоит по умолчанию слать `think: false`, включая thinking только по
  явному запросу пользователя.

## 6. Доп. умение: диалоговый режим `/ai` (тред через reply в Telegram)

Пользователь отдельно попросил спроектировать (не реализовывать —
реализация тоже за другой сессией) диалоговый UX поверх `llm`:

- `/ai [текст]` — новое сообщение с этой командой **всегда** начинает
  **новый** диалог, независимый от предыдущих. Текст после команды —
  опционален.
- Бот отвечает сообщением с ответом нейросети (или, если текста не было —
  сообщением-заглушкой «диалог начат, жду ваше сообщение»).
- Если пользователь отвечает (**Telegram reply**, `reply_to_message`) на
  **это** сообщение бота (или на любое последующее сообщение бота в этом
  же треде) — это продолжение диалога: LLM получает всю историю треда, а
  не только последнюю реплику.
- Если пользователь пишет **не как reply** (обычное сообщение) — это
  **не** воспринимается как продолжение диалога, вообще не трогается этой
  фичей.

Технически это стандартный и несложный паттерн (Telegram сам даёт
`message.reply_to_message.message_id` — дерево реплаев строить не нужно,
хранить нужно только плоскую таблицу «сообщение → к какому диалогу
относится»). Ниже — конкретный дизайн под текущий код репозитория
(проверено чтением `db/schema.sql`, `bot/handlers/control.py`,
`bot/service_link.py`, `bot/actions.py`, `bot/commands.py`).

### 6.1 Схема БД (добавить в `db/schema.sql`)

```sql
-- Диалоги /ai: плоская таблица «сообщение → диалог». dialogue_id —
-- message_id самой команды /ai, с которой начался тред (не отдельный uuid
-- — свой же способ адресации уже монотонен и уникален в рамках чата).
-- PRIMARY KEY составной: telegram message_id уникален только в рамках
-- одного чата, не глобально (при нескольких подписках/чатах коллизия
-- иначе неизбежна).
CREATE TABLE IF NOT EXISTS ai_turns (
    chat_id       INTEGER NOT NULL,
    message_id    INTEGER NOT NULL,   -- id ИМЕННО этого сообщения (юзера или бота)
    dialogue_id   INTEGER NOT NULL,   -- message_id команды /ai, начавшей тред
    role          TEXT NOT NULL,      -- 'user' / 'assistant'
    content       TEXT NOT NULL,
    created_at    TEXT NOT NULL,
    PRIMARY KEY (chat_id, message_id)
);
CREATE INDEX IF NOT EXISTS idx_ai_turns_dialogue ON ai_turns(chat_id, dialogue_id, message_id);
```

### 6.2 Поток `/ai [текст]` (новый диалог)

1. `dialogue_id := message.message_id` (id самого сообщения с `/ai`).
2. Если текст есть:
   - записать `ai_turns(chat_id, message_id=dialogue_id, dialogue_id, role='user', content=текст)`;
   - вызвать `llm.chat` (см. §6.4) с `messages=[{"role":"user","content":текст}]`;
   - отправить ответ бота (`reply_to_message_id = dialogue_id` — красиво в UI, хотя логика треда на это не полагается);
   - записать `ai_turns(chat_id, message_id=<id отправленного сообщения бота>, dialogue_id, role='assistant', content=ответ)`.
3. Если текста нет:
   - отправить сообщение-заглушку «🤖 Диалог начат. Ответьте на это сообщение, чтобы продолжить.» (без обращения к LLM — не тратить впустую вызов);
   - записать её как `ai_turns(chat_id, message_id=<id заглушки>, dialogue_id, role='assistant', content='')` — пустой content нужен, чтобы при первом reply история для LLM корректно оказалась пустой (не пытаться "объяснить" LLM пустую заглушку).

### 6.3 Поток продолжения (reply на сообщение бота из треда)

Фильтр хендлера: `message.reply_to_message is not None`, и по
`(chat_id, reply_to_message.message_id)` в `ai_turns` находится строка —
если не находится (реплай на что-то другое) — эта фича молчит, событие
не трогает (обычный пропуск, другим хендлерам/no-op). Точный механизм
фильтра (aiogram `Filter` с доступом к `Store`, либо ранний `return` внутри
хендлера после проверки) — на усмотрение реализации, оба варианта
одинаково корректны.

Если строка найдена:
1. `dialogue_id := найденная_строка.dialogue_id`.
2. Собрать всю историю: `SELECT role, content FROM ai_turns WHERE chat_id=? AND dialogue_id=? ORDER BY message_id` (пропустить пустые content от заглушки).
3. Добавить новую реплику пользователя в конец списка (`role='user'`), записать её в `ai_turns` (`message_id = message.message_id`, тот же `dialogue_id`).
4. Вызвать `llm.chat` со всей историей.
5. Отправить ответ (`reply_to_message_id = message.message_id`), записать как `ai_turns(message_id=<id ответа>, тот же dialogue_id, role='assistant')`.

Тред продолжается сколько угодно раз — каждое новое сообщение бота само
становится валидным якорем для следующего reply, т.к. тоже попадает в
`ai_turns` с тем же `dialogue_id`.

### 6.4 Действие `chat` в службе `llm` (расширение §2.1)

Однократного `ask(prompt)` (§2.1) для этого недостаточно — нужна история.
Добавить action `chat`:
```json
{"id": "chat", "title": "Диалог с нейросетью", "params": [
  {"name": "messages", "type": "json", "required": true, "title": "История сообщений"}
]}
```
`args["messages"]` — список `{"role": "user"|"assistant", "content": str}`
(без обёртки/строкового кодирования — payload конверта и так произвольный
JSON, `proto/server.py._run_command` проверяет только *наличие*
обязательных параметров, не строгую схему типа — см. `type` в
`ActionParam` как чисто описательное поле для UI, не enforced validator).
Внутри `LlmService.run_command("chat", args)` — POST на
`http://127.0.0.1:11434/api/chat` (не `/api/generate` — chat-эндпоинт сам
знает шаблон чата модели, включая system-промпт и роли) с
`{"model": ..., "messages": args["messages"], "stream": false, "think": false}`.
`think` по умолчанию `false` (см. заметку в §5 про задержку thinking-режима)
— можно вынести в конфиг `[llm]`, если захочется управлять этим позже.

### 6.5 Права и меню

Новая команда `ai` в `bot/commands.py` (по образцу `SCAN_NOW`): `universal=False`,
`menu=True` (или `False` — на усмотрение, это довольно «первоклассное»
умение роя, вероятно стоит в меню), `right="chat@llm"` — обычный `<action>@<служба>`
паттерн, уже используемый для `apps`/`monitor`. Обработчик продолжения
(reply) отдельного права не требует — он не новая команда, а реакция на
уже разрешённый ранее диалог (право проверяется один раз, при `/ai`).

### 6.6 На заметку реализации (не блокеры, но стоит учесть)

- **Таймаут ответа.** См. §3 — `chat` с длинной историей может отвечать
  дольше, чем `ask`. Плюс стоит вызвать `bot.send_chat_action(chat_id,
  "typing")` перед вызовом `llm.chat` — иначе 10-30-секундная пауза в
  чате выглядит как зависший бот.
- **Рост контекста.** Дефолтный контекст Ollama на этой ноде — 4096
  токенов (авто-подбор от VRAM, см. §1). Очень длинный тред `/ai`
  когда-нибудь упрётся в это; либо резать историю (например, последние N
  реплик) перед отправкой в `chat`, либо поднять `OLLAMA_CONTEXT_LENGTH` в
  конфиге контейнера ollama — что важнее, решать по месту.
- **Не блокирующая уборка.** `ai_turns` никогда не чистится — можно со
  временем добавить в существующий `housekeeping_cron` (тот же паттерн,
  что уже применяется в scheduler'е для других задач) прунинг диалогов
  старше N дней. Не нужно для MVP.
- **System-промпт.** В сессии обсуждали анти-нейромлоп системный промпт
  для более «человеческих» ответов — решение, включать ли его по
  умолчанию для `/ai` (и в каком виде — глобально в конфиге `[llm]` или
  оставить чистым), это продуктовое решение бот-стороны, не навязываю.

## 7. Инфраструктура tool-calling (фундамент для §8–9)

Сейчас `llm/ollama.py::chat()` зовёт `/api/chat` без `tools` — модель
только отвечает текстом. Чтобы Альфред мог сам решать «дёрнуть тул или
ответить», нужно прокинуть параметр `tools` (формат — как в OpenAI
function-calling, Ollama его поддерживает нативно для tool-calling-моделей,
`qwen3:14b` в их числе) и обработать `message.tool_calls` в ответе.

### 7.1 Ключевое архитектурное решение: диспетчер тулов — в боте, не в службе `llm`

Сегодня **только фронтенд** (`bot/`) умеет адресоваться к произвольной
службе на произвольной ноде (`Address{node, service}`, `swarm_view`,
`wake_swarm_node_core` и т.п.). Ни одна служба (`apps`/`monitor`/`llm`)
сама не ходит по протоколу к другим службам — только фронтенды к ним, и
`node/service.py` для внутренней маршрутизации пиров (см. проверено
`grep ProtoClient` — совпадений в бизнес-службах нет). Заводить это же
умение внутри `LlmService` означало бы дублировать роутинг, который уже
есть у бота, ради одной службы.

Значит:
- `llm/ollama.py::chat()` — принимает опциональный `tools: list[dict] | None`.
- `LlmService.run_command("chat", …)` — принимает опциональный
  `args["tools"]` (список деклараций, JSON), пробрасывает в `ollama.chat`.
  Если Ollama вернула `tool_calls` вместо готового текста — служба **не
  пытается их исполнить**, а отдаёт как есть:
  `{"tool_calls": [...], "model": ...}` вместо `{"response": ...}`.
- Цикл «вызвать тул → вернуть результат модели → получить либо новый
  tool_call, либо текст» живёт в `bot/ai_flow.py` (там же, где уже есть
  вся оркестрация `/ai` — presence/wake, `_reply_context_lines` и т.д.):
  1. Собрать `messages` как сегодня + статичный `tools` (объединение
     деклараций из §8/§9) → `chat`.
  2. Если пришёл `response` — как сегодня, отправить и записать в `ai_turns`.
  3. Если пришли `tool_calls` — для каждого найти обработчик по имени в
     явном реестре `TOOL_HANDLERS: dict[str, Callable]` (не общий
     прокси «дёрни любую службу по имени из аргументов модели» — это была
     бы дыра в правах; каждый тул — узкая, руками прописанная функция).
     Выполнить, получить результат.
  4. Дописать в `messages`: `{"role": "assistant", "tool_calls": [...]}`,
     затем `{"role": "tool", "content": <результат как JSON-строка>}` на
     каждый вызов, повторить `chat`.
  5. Ограничить число раундов (например, 3) — защита от зацикливания
     модели на бесконечных tool_call без финального ответа; при
     превышении — тот же путь, что и прочие внутренние сбои (`ALBERT_HICCUP`
     персонаж, не голая ошибка).

### 7.2 Права и безопасность

Тулы дёргает сама модель, не пользователь явной командой — значит нельзя
давать через них ничего, что пользователь и так не может. Правило:
**в реестр `TOOL_HANDLERS` попадают только read-only действия**, и он один
и тот же для всех, у кого уже разрешён `/ai` (право `chat@llm`) — тулы не
самостоятельное право, а часть того, что открывает `chat@llm`. Никаких
`start`/`stop`/`restart` и вообще ничего пишущего в этом реестре не
может быть в принципе (см. §8.5 про напоминания — это единственное
исключение, и то запись только в собственную узкую табличку, не общее
управление системой).

### 7.3 Отказы тулов не должны ломать персонажа

Нода спит/недоступна/таймаут — обычная деградация роя (ARCHITECTURE §11
п.4), не исключение уровня Python наружу. Обработчик тула такие случаи
ловит сам и возвращает **текстовый** результат вида `"недоступно: нода
winpc спит"` — это отдаётся модели как обычный результат tool-вызова, а
не как сбой цикла; персонаж дальше сам решает, как сформулировать отказ
(system-промпт и так требует не выдавать технических деталей). Настоящим
сбоем (`ALBERT_HICCUP`) остаётся только реальная поломка самого цикла
`chat` (Ollama упала и т.п.), не «тул не дозвонился до соседней ноды».

### 7.4 Таймаут

Цикл в несколько раундов (генерация → тул → снова генерация) может уйти
дальше, чем сегодняшний `request_timeout_s` для одного `chat`. Нужен
отдельный, более широкий таймаут на весь тред `/ai` с тулами в
`bot/ai_flow.py` (не трогать `LlmConfig.request_timeout_s` — это таймаут
одного обращения к Ollama, он и так работает верно).

## 8. Скиллы первой очереди (без SearXNG — см. §9 отдельно)

Порядок — от того, что вообще не требует §7, к тому, что требует его
полностью. Делать в этом порядке: каждый следующий пункт добавляет
минимум одну новую движущуюся часть.

### 8.1 Текущее время — БЕЗ tool-calling вообще

Маленькие локальные модели плохо знают «сейчас». Но это не повод заводить
тул: дата/время нужны практически на каждый вызов, а не иногда — дешевле
и надёжнее просто дописывать строку в контекст на каждый `/ai`-запрос,
рядом с тем, что `_reply_context_lines` уже делает для reply/quote:
```python
f"Точное время сейчас: {datetime.now(ZoneInfo('...')):%Y-%m-%d %H:%M}, {weekday_ru}."
```
Ноль новой инфраструктуры, ноль сетевых обращений, работает уже сегодня.
**Сделать первым, отдельно от §7** — не зависит от tool-calling вообще.

### 8.2 Калькулятор — тул без сети и без роя

Маленькие модели слабы в арифметике. Тул `calc(expression: str)`,
обработчик — целиком внутри бота (никакого прото-вызова никуда): парсинг
через `ast` с белым списком узлов (числа, `+-*/`, `**`, скобки — **не**
`eval()` голого выражения, инъекция кода недопустима) → число или
`"ошибка: некорректное выражение"`. Хорош как первый реальный тест цикла
из §7 — если он поедет, остальные тулы (уже сетевые/по рою) будет
проверять проще, изолировав баг «сломан цикл» от бага «сломан конкретный
тул».

### 8.3 Read-only статус роя — первый тул поверх реального `ProtoClient`

Тул `swarm_status(node: str | None, what: "health"|"disks"|"uptime"|"torrents")` —
обработчик в боте делает то же самое, что уже делают
`bot/handlers/status.py`/`swarm_view.py` для команд `/status`/`/node`, только
результат not рендерится в текст для Telegram, а возвращается моделью как
JSON. Реюз существующего кода рендеринга/сбора состояния, не новый путь к
данным — просто новый потребитель тех же `get_state()`.

### 8.4 Погода — первый тул с внешним (не-swarm) API

Open-Meteo, без ключа, координаты дома — фиксированные в конфиге. По
аналогии с правилом «фронтенд в систему не ходит» (см. `apps/service.py`
docstring) — сетевой вызов делает не бот напрямую, а тонкая служба на
alfred (по образцу `llm`/`ollama.py`: тот же `urllib` + `asyncio.to_thread`,
без новых HTTP-зависимостей). Можно назвать `net` — задел на то, что §9
(поиск) ляжет туда же вторым действием, а не отдельной третьей службой.

### 8.5 Напоминания — единственный «пишущий» тул, и то не совсем

> **УСТАРЕЛО (см. блок "Статус" перед §7, сессия 2026-07-24, вечер)** —
> раздел ниже описывает первоначальный дизайн ("узкая табличка в БД
> бота"). Реализовано было иначе: отдельный сервис роя `tasks`, живой
> ответ модели в момент срабатывания, self-scheduling.

`remind(when: str, text: str)` — по образцу существующего
`housekeeping_cron`/scheduler-паттерна. Формально не read-only (пишет
строку), но пишет только в свою узкую табличку «отложенное сообщение
самому себе», не управляет системой — риск тот же, что у обычной
пользовательской команды, не выше. Можно отложить после 8.1–8.4, если
хочется сначала закрепить сам цикл tool-calling на простых read-only
кейсах.

## 9. Веб-поиск через self-hosted SearXNG (отдельный, более поздний трек)

Обсуждали отдельно: Tavily/Brave не подходят пользователю, решили —
локальный SearXNG. Делать **после** §8, а не вместо — это самый тяжёлый
из тулов по количеству новых частей (отдельный процесс, Redis, конфиг
движков), остальные скиллы должны обкатать сам цикл §7 первыми.

- **Нода: alfred**, не winpc — alfred единственная 24/7 headless-нода
  (winpc спит/перезагружается под игры, Tailscale там поднимается только
  после интерактивного логина — см. §5 выше), и не стоит утяжелять и без
  того хрупкий GPU-стек winpc (WSL2 + Docker) ещё одной, не относящейся к
  инференсу движущейся частью.
- **Установка без Docker** — на alfred docker не стоит (см. диагностику
  железа CLAUDE.md), тащить его ради одного лёгкого сервиса избыточно.
  Ставить через официальный неdocker-путь SearXNG (venv + uWSGI,
  `utils/searxng.sh`), под systemd-юнитом, слушает `127.0.0.1:8888`.
- **`format=json` выключен по умолчанию** в `settings.yml` — включить
  явно, иначе служба-обёртка получит HTML вместо структурированного
  ответа.
- **Движки** — выбрать те, что не банят скрейпинг без ключа (DuckDuckGo,
  Brave, Startpage, Mojeek); Google в SearXNG быстро упирается в капчу,
  не включать.
- **Redis — чисто эфемерный кэш**, без AOF, RDB редко/выключен: на
  alfred отдельно отслеживается износ eMMC (сейчас здорова, 0–10%,
  ~6.4GB lifetime writes — см. CLAUDE.md), лишние постоянные записи ради
  кэша, потеря которого не страшна при рестарте, — не нужны.
- **Служба-обёртка роя** — новое действие `search(query: str)` в той же
  тонкой сетевой службе, что и погода (§8.4, условно `net`): urllib GET на
  `http://127.0.0.1:8888/search?q=...&format=json`, вернуть top-N
  `{title, url, snippet}` — без скрейпинга самих страниц по ссылкам,
  только то, что уже отдал SearXNG.
- Право/тул `search` — в тот же реестр `TOOL_HANDLERS` из §7.2 (read-only,
  доступен всем, кому разрешён `/ai`).
