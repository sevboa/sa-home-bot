# Авторизация команд sa-home-bot

Модель прав перенесена из рабочего бота, упрощена под домашний контур (нет
хаб-команд и отчётов). Архитектура — в [`ARCHITECTURE.md`](./ARCHITECTURE.md).

## 1. Модель

Авторизация — **на уровне чата (chat-level)**, не пользователя. Любой участник
чата имеет права, заданные для этого чата в конфиге. Персональных прав нет: если
человеку нужны личные права — он заводит личный чат с ботом и отдельную подписку
для своего `chat_id`.

## 2. Источник прав

Права чата = поле `allowed_commands` его подписки в `config.toml`. Конфиг
иммутабелен во время работы; изменение прав = правка конфига + рестарт.

## 3. Категории команд

Две категории (хабов, в отличие от рабочего бота, нет).

### 3.1 Универсальные — всегда и везде, без проверок

- `/help` — список доступных в этом чате команд;
- `/ping` — проверка живости (pong);
- `/whoami` — `user_id` и `chat_id` текущего чата (чтобы узнать значения для
  конфига).

Не перечисляются в `allowed_commands`, middleware их не проверяет.

### 3.2 Управляющие — требуют явного разрешения

Выполняются только если: чат подписной, не `broken`, и имя команды есть в
`allowed_commands` его подписки.

- `/qbittorrent`, `/jellyfin`, … — динамические скилы-приложения из `describe`
  службы apps (команды меню первого уровня); право — `<id>@apps` (§3.3),
  проверяется хендлером, т.к. реестр команд про них не знает;
- `/nodes` — список нод; открывает иерархию: карточка ноды (право `status`) →
  карточка службы (право `nodes`) → кнопки управления (права `…@node`);
- `/status` — карточка локальной ноды (мониторинг CPU/дисков + службы),
  scope чата;
- `/stats` — статистика прогонов сканера;
- `/scan_now` — форс-скан (ставит job в очередь);
- `/wake` — разбудить внешнюю машину по Wake-on-LAN (секция `[wake]` конфига);
  та же операция — кнопка «Разбудить ПК» в списке нод (право `wake`);
- `/alfred` — диалог с Альфредом (LLM-служба `llm`, обычно на отдельной
  ноде); `/ai` — тот же скилл под тем же правом, скрытый алиас, не в меню/
  /help (как `/swarm`↔`/nodes`, сознательно — чтобы в общем чате не так явно
  читалось как ИИ). Реплай на сообщение бота из уже начатого треда продолжает
  диалог без повторной проверки прав (право проверяется один раз, на
  `/alfred`/`/ai`). Право — **`chat@llm`**, буквально в форме «действие@служба»
  (§3.3), хотя это обычная команда бота (`Command.right`, не динамическая
  кнопка из `describe`): так право за одно умение не завязано на два разных
  механизма проверки;
- (этап 2) `/mute`, `/unmute` — заглушить алерты по компоненту на время.

Если хоть одно условие не выполнено — короткий ответ «Команда недоступна в этом
чате», действие не выполняется, в другие чаты ничего не шлётся.

### 3.2.1 Алиасы и динамические команды навигации

- `/swarm` и `/nodes` — одна сводка роя под одним правом **`nodes`**
  (поле `Command.right`): алиас не требует правки конфигов.
- `/node_<id>` (карточка ноды) и `/svc_<нода>_<служба>` (карточка службы) —
  динамические ссылки-команды, которых нет в реестре: AuthorizationMiddleware
  их пропускает, право проверяет обработчик (`bot/handlers/node_links.py`):
  `/node_*` — право `status` (как карточка ноды), `/svc_*` — `nodes`
  (как карточка службы). Дефисы в именах нормализуются в подчёркивания
  (`arch-t480` → `/node_arch_t480`), обратный разбор — сопоставлением с
  реальным составом роя.

### 3.3 Динамические действия служб — право `действие@служба`

Кнопки действий бот строит из `describe` служб (протокол v0), ничего не
хардкодя: id, название и параметры приходят от службы. Право на такую кнопку —
запись **`действие@служба`** в `allowed_commands`, например:

- `scan_now@monitor` — кнопка «Скан» на карточке ноды (`/status`);
- `start@node`, `stop@node`, `restart@node` — кнопки управления на карточке
  службы;
- `qbittorrent@apps`, `jellyfin@apps` — скилы-приложения: команда в меню
  бота (состояние юнита + ссылка на веб-морду);
- `start@apps`, `stop@apps`, `restart@apps` — кнопки реального управления
  systemd-юнитом на карточке приложения (нужны настроенные права на машине —
  `nodectl fix`, см. ARCHITECTURE/PROTOCOL про `needs_privilege`).
- `check_update@node`, `update@node` — самообновление ноды через pipx (root
  не нужен); умения объявляются, только если нода поставлена из git-репо
  через pipx (dev-чекаут — не объявляются вовсе). `update` не перезапускает
  процесс — рестарт делает человек через `restart_node`, см. `node/update.py`.

Голое имя действия (`scan_now`) тоже принимается — совместимость со старыми
конфигами (даёт право на действие у любой службы). С появлением удалённых нод
форма расширится до `действие@служба@нода`.

## 4. Scope

Любая управляющая команда влияет **только на текущий чат**. Глобальных
модификаций нет. `/scan_now` запускает общий скан (физически глобальный
эффект), но право давать команду — chat-level.

## 5. Middleware

`AuthorizationMiddleware` проверяет права до handler'а:

```
1. chat_id из апдейта.
2. Универсальная команда → пропустить без проверок.
3. Управляющая:
   a) найти подписку по chat_id; нет → отказ;
   b) broken → отказ;
   c) имя команды в allowed_commands? нет → отказ;
   d) иначе → пропустить, инжектить подписку и chat_id в handler.
```

## 6. Меню Telegram (`set_my_commands`)

При старте, после `validate_on_startup`:

- **default scope** — только универсальные;
- **per-chat scope** — для каждой не-broken подписки: универсальные +
  управляющие из её `allowed_commands`.

`set_my_commands` — это **UX, не security**: пользователь может отправить любую
команду текстом; реальная проверка — в middleware.

## 7. `/help`

Универсальна, но содержимое зависит от чата: в подписном — универсальные +
доступные управляющие; в неподписном — только универсальные. Недоступные команды
не упоминаются. Тексты описаний — из единого реестра `bot/commands.py` (без
дублирования).

## 8. Инварианты

1. Авторизация только chat-level; user-level нет.
2. Универсальные (`/help`, `/ping`, `/whoami`) — всегда и везде, не в
   `allowed_commands`.
3. Управляющие — только в подписном не-broken чате с правом в `allowed_commands`.
4. `/help` показывает только реально доступные команды.
5. `set_my_commands` — UX; защита — в middleware.
6. Меню обновляется при каждом старте после валидации подписок.
7. Scope любой управляющей команды — текущий чат.
8. Broken-подписки не позволяют выполнять управляющие команды.
9. Описания команд — в одном месте (`bot/commands.py`).

## 9. Сознательно не делается

- Per-user права в одном чате.
- Команды делегирования прав (`/grant`, `/revoke`).
- Глобальный «суперадмин».
- Динамическое изменение `allowed_commands` через бота.
- Хаб-команды (нет отчётов/витрин в MVP).
